# install.sh 改善設計

## 背景

現在の `install.sh` は、リポジトリ内の各トップレベルディレクトリを手動で `~/.config/` にシンボリックリンクしている。ディレクトリごとのリンクの場合、`.config/nvim/` などにリポジトリ外のプログラムが生成するファイル（`lazy-lock.json` やプラグインディレクトリなど）が混入すると、それらが意図せずバージョン管理対象になってしまうリスクがある。

## 目的

- 手動での `ln -sf` 追加を不要にする
- ディレクトリ単位のシンボリックリンクをやめ、**ファイル単位でシンボリックリンク**を張り、ディレクトリ自体は実体で作成する
- リポジトリに追加された新しいファイル・ディレクトリが自動的に反映される

## アーキテクチャ

```
.dotfiles/
├── install.sh          ← 自動検出・自動リンク
├── .bashrc             → ~/(.bashrc)
├── ghostty/
│   └── config.ghostty  → ~/.config/ghostty/config.ghostty
├── nvim/
│   └── init.lua        → ~/.config/nvim/init.lua
├── tmux/
│   └── tmux.conf       → ~/.config/tmux/tmux.conf
└── ...
```

## ターゲット分類（自動検出）

スクリプトはリポジトリルートの内容を自動検出する。以下は現在の例。

1. **ルートファイル**（`~/<filename>` に直接リンク）
   - `.bashrc`
   - 上記除外ルールに該当しない、ルートに配置されたその他のファイルも自動的にリンク対象になる
2. **トップレベルディレクトリ**（`~/.config/<dirname>/` 下に展開）
   - `ghostty/`, `lazygit/`, `nvim/`, `opencode/`, `tmux/`, `tmuxp/`, `yazi/`
   - 上記除外ルールに該当しない、ルートに配置されたその他のディレクトリも自動的にリンク対象になる

## 展開方式

- `find` で各トップレベルディレクトリ内のファイルを再帰的に列挙する
- 各ファイルの相対パスを `~/.config/<dirname>/<relpath>` に個別シンボリックリンクする
- リンク先の親ディレクトリが存在しない場合は `mkdir -p` で実体を作成する
- 既存のリンクが正しく指している場合はスキップする
- 誤ったリンクや通常ファイルの場合は上書き（`-f`）する

## 除外ルール

- `.git/`, `.worktrees/`
- リポジトリ管理ファイル: `install.sh`, `AGENTS.md`, `.gitignore`
- `.gitignore` で無視されているファイル・ディレクトリ（`git check-ignore` で判定）
- `*.md` は除外しない

## 実行環境・エラーハンドリング

- 純粋な Bash スクリプトで実装（外部依存なし）
- `set -euo pipefail` で厳密モード
- スクリプトの場所を自動検出（`BASH_SOURCE`）して `DOTFILES_DIR` を決定

## 採用アプローチ

拡張シェルスクリプト（A案）。外部ツール不要で、dotfiles のセットアップ時に追加インストールが発生しないため。
